# Demo App Design: Support Ticket System

## Purpose

A minimal support ticket app used to record SDD demo videos for the WebConf 2026 presentation. The app must look credible on camera while being trivial to set up and run. No real backend — MSW provides all data.

The baseline app ships with a working ticket list + detail view. The **demo feature** (added during the SDD recordings) is a search filter with debounced input.

## Data Model

```ts
interface Ticket {
  id: string
  title: string
  description: string
  status: "open" | "in-progress" | "resolved"
  priority: "low" | "medium" | "high"
  createdAt: string   // ISO 8601
  updatedAt: string   // ISO 8601
  assignee: string
}
```

MSW seeds 15-20 tickets with realistic support content (password resets, billing issues, API errors, feature requests, etc.).

## API Endpoints (MSW)

| Method | Path               | Response                    |
|--------|--------------------|-----------------------------|
| GET    | /api/tickets       | `Ticket[]`                  |
| GET    | /api/tickets/:id   | `Ticket`                    |

Both return JSON. No pagination needed for the seed data size.

## Component Details

### Layout.tsx
- Chakra `Flex` horizontal split: left (60%) and right (40%)
- Header bar with "Acme Support" text and a subtle background color
- Full viewport height

### TicketList.tsx
- Chakra `Table` component
- Columns: title, status (Badge), priority (Badge), created date (relative or short format)
- Clicking a row calls `onSelect(ticketId)` — selection state lives in parent or URL search param
- Selected row gets a highlight background
- Loading state: Chakra Skeleton rows
- Status badge colors: open=blue, in-progress=yellow, resolved=green
- Priority badge colors: low=gray, medium=orange, high=red

### TicketDetail.tsx
- Receives `ticketId: string | null`
- When null: centered "Select a ticket to view details" message
- When set: fetches ticket via `useTicket(id)`, displays all fields
- Chakra Card with Stack layout for fields

## TanStack Query Hooks

### useTickets()
```ts
useQuery({
  queryKey: ["tickets"],
  queryFn: () => fetch("/api/tickets").then(r => r.json()),
})
```

### useTicket(id)
```ts
useQuery({
  queryKey: ["tickets", id],
  queryFn: () => fetch(`/api/tickets/${id}`).then(r => r.json()),
  enabled: !!id,
})
```

## MSW Setup

- Browser mode only (`setupWorker`)
- Started conditionally in `main.tsx` before React renders (await `worker.start()`)
- Handlers use an in-memory `tickets` array from `data.ts`
- `mockServiceWorker.js` generated via `npx msw init public/`

## Seed Data

15-20 tickets with varied:
- Statuses: mix of open (majority), in-progress, resolved
- Priorities: mix of all three
- Titles: realistic support topics (e.g., "Cannot reset password", "API rate limit errors on dashboard", "Billing shows wrong plan", "Feature request: dark mode")
- Assignees: 3-4 fictional names
- Dates: spread over the last 2 weeks

## What's NOT in the Baseline

These are intentionally excluded — the search filter is the demo feature:

- **No search/filter** — added during SDD demo recording
- **No create/edit/delete** — not needed for demos
- **No auth** — as specified
- **No dark mode** — default Chakra light theme
- **No tests** — the demo may add tests as part of the SDD workflow
- **No deployment** — local dev only
